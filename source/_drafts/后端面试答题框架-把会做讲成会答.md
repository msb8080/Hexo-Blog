## TL;DR

1. 面试答题核心不是“背更多点”，而是“让结论、逻辑、案例可验证”。
2. 固定答题框架可以显著降低紧张状态下的表达波动。
3. 每道题都要覆盖边界、方案、权衡、真实实践四个层次。

## 背景

- 常见失误是“知道很多，但讲不成结构”，导致面试官难以判断真实能力。
- 需要一个可重复训练的答题模板，把技术理解转化为可评估表达。

## 目标

- 建立统一答题框架，覆盖基础题、设计题、项目题。
- 降低答题跑偏和信息堆砌，提升回答完成度。
- 形成可复盘的训练机制（录音、改稿、二次表达）。

## 通用答题框架（5 步）

1. 定义边界：明确问题范围和前提条件。
2. 先给结论：用一句话回答核心判断。
3. 讲落地方案：说明如何在工程里实现。
4. 给真实案例：用项目经历证明可行性。
5. 说权衡风险：成本、复杂度、回滚策略。

## 题型映射模板

| 题型 | 优先讲什么 | 常见误区 |
|---|---|---|
| 基础原理题 | 定义 + 结论 + 场景 | 只背概念，不讲应用 |
| 架构设计题 | 约束 + 方案 + 权衡 | 直接画大图，忽略约束 |
| 故障复盘题 | 现象 + 根因 + 处置 + 预防 | 只讲结果，不讲过程 |

## 示例题演示（模板）

### 1) 如何设计接口限流策略？

- 结论：先按用户维度限流，再按接口热点分层限流。
- 落地：网关限流 + 业务侧令牌桶 + 熔断降级。
- 案例：在高峰期将错误率从 X 降到 Y（待补真实数据）。
- 权衡：阈值过紧会误伤正常请求，需要灰度调参。

### 2) 缓存和数据库一致性怎么处理？

- 结论：优先保证最终一致性，避免强一致带来写入瓶颈。
- 落地：延迟双删、重试队列、幂等更新。
- 案例：某查询接口读延迟优化（待补真实过程）。
- 权衡：实现复杂度提升，需要配套监控和补偿机制。

### 3) 你做过最有价值的一次性能优化是什么？

- 结论：先定位瓶颈，再做最小改动，最后用指标验证。
- 落地：火焰图/慢日志定位 -> 索引或缓存优化 -> 观察 P95。
- 案例：优化前后对比表（待补）。
- 权衡：短期见效方案和长期架构方案要分阶段推进。

## 训练与复盘机制

1. 每周选择 5 题录音，限制单题 3 分钟。
2. 按“边界/结论/方案/案例/权衡”逐项打分。
3. 对低分题做二次改稿，48 小时内复答。
